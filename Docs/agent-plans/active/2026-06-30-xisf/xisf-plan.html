Overview

The goal is to build SwiftXISF, a Swift library that parses XISF (Extensible Image Serialization Format) files, replicating the architecture, naming, file organization, conventions, and testing approach of the sibling SwiftFITS project as closely as possible. The public API should feel like a natural counterpart to SwiftFITS: an XISFFile entry point opened from a URL or Data, exposing parsed images, properties, FITS keywords, and metadata, with image pixel data surfaced as opaque decompressed bytes plus typed geometry/format metadata (interpretation left to the consumer, exactly as SwiftFITS leaves data segments opaque).

Unlike FITS (fixed 2880-byte ASCII blocks), an XISF monolithic file is a small binary preamble + a UTF-8 XML header describing the unit, followed by attached binary data blocks. The parser therefore splits naturally into three layers: a monolithic-file container reader (signature, header length, attached-block region), an XML header model (elements parsed with Foundation's XMLParser into typed Swift values), and a data-block pipeline (location resolution → decompression + byte un-shuffling → checksum verification → opaque bytes). The plan is staged so that each layer is built and tested before the layers that depend on it, and every milestone is independently reviewable and committable.

Scope decisions (see Q&A): the library is read-only (no XISF authoring); pixel data is opaque bytes + metadata; and the optional features — checksum verification, external/distributed data blocks, color/ICC & thumbnail metadata, and zstd (deferred, as the one external dependency) — are all in scope as their own milestones.

Questions & answers

Should SwiftXISF be a read-only parser, or also support writing/encoding XISF files?
Read-only parser. Focus on decoding/parsing only, matching the source doc's stated goal. It may still expose raw bytes, but there is no XISF authoring/serialization API.
How should pixel/image data be exposed?
Opaque bytes + metadata. Mirror SwiftFITS: expose the fully decompressed and un-shuffled raw block bytes plus typed metadata (geometry, sampleFormat, colorSpace, pixelStorage, byteOrder, bounds). The consumer interprets samples. No decoding into typed Swift sample arrays.
Which optional XISF features should be in scope?
All four. (1) zstd as a deferred later milestone (introduces an external dependency). (2) Data-block checksums (sha-1 / sha-256 / sha-512 and other supported digests). (3) External/distributed files (url(...) / path(...) data-block locations). (4) Color/ICC & thumbnails (ICC profiles, RGB working space, display functions, thumbnail images).

Baseline assumed in every milestone regardless of the above: monolithic files; inline / embedded / attachment-within-file data blocks; multiple images per file; XISF properties; embedded FITS keywords; zlib + lz4 / lz4hc compression with byte-shuffling.

Notes

Reference projects on disk. SwiftFITS is checked out as a sibling at ../SwiftFITS relative to this repository. Read it to replicate structure and style, but never reference absolute SwiftFITS paths from SwiftXISF code or tests (per the source doc). The verified XISF 1.0 structural facts used to write this plan are collected in the Appendix.

SwiftFITS conventions to replicate (verified by reading ../SwiftFITS):

  • One public type per file, PascalCase file names matching the type (XISFFile.swift, XISFImage.swift, …).
  • MIT license banner at the top of every file (Jean-David Gadina — www.xs-labs.com, year 2026), followed by import Foundation. The existing SwiftXISF/SwiftXISF.swift and Package.swift already carry this banner.
  • Access control: explicit public on the API surface; public private(set) for externally-readable, internally-set state; internal/private for parsing helpers. (Matches both SwiftFITS and the user's library conventions.)
  • SwiftDoc triple-slash comments on every public and private type/member, including Sendable/thread-safety notes on reference types.
  • Errors: a single XISFError enum conforming to LocalizedError, CustomStringConvertible, Sendable with associated values carrying context (URL, offsets, reasons).
  • Parsing strictness: an XISFParsingOptions OptionSet mirroring FITSParsingOptions (lenient vs. strict handling of spec deviations, optional checksum verification, whether external/distributed locations may be resolved).
  • Foundation extensions for low-level helpers (Data, String, CharacterSet), one per file, matching SwiftFITS.
  • Testing: swift-testing (import Testing, struct Test_<Type>, @Test, #expect/#require). Real sample files live in a top-level Test Files/ directory outside the package target and are loaded via #filePath under SPM (with a Bundle.module fallback) — exactly as SwiftFITS does. The existing SwiftXISFTests/SwiftXISFTests.swift placeholder will be removed.
  • Xcode project parity: SwiftXISF.xcodeproj exists and must be kept in sync with Package.swift as files are added (per the user's Xcode-project convention). Prefer building/testing via xcodebuild.

Open decision — folder organization. SwiftFITS keeps all ~10 source files flat at the target root. SwiftXISF will have noticeably more types (~25+). The plan therefore proposes grouping sources into subfolders by concern (Core/, Model/, DataBlock/, Compression/, Color/, Extensions/), which aligns with the user's "organize into subfolders when necessary" convention while keeping every other SwiftFITS convention identical. Confirm at M1 whether to use subfolders or stay flat to match SwiftFITS exactly.

Open decision — zstd dependency (M13). SwiftFITS has no external dependencies. Apple's Compression framework covers zlib and lz4/lz4hc (LZ4_HC produces standard LZ4 streams, decodable by the LZ4 decoder), but not zstd. zstd is therefore isolated in the final milestone and requires choosing/approving an external dependency (e.g. a SwiftPM zstd wrapper or vendoring libzstd). This dependency choice must be confirmed before M13 is implemented.

Key implementation references: byte-level reads use little-endian for the header-length field and honor each image's byteOrder attribute for sample data (default big-endian per spec — to be verified during M9). Decompression uses Apple's Compression framework; checksums use CryptoKit (SHA-256/512) with CommonCrypto/Insecure SHA-1 for sha-1, and SHA-3 only if a practical implementation is available.

Phase 2 · Monolithic container & XML header
M3

Monolithic file container

Completed

Goal: Parse the binary preamble and locate the XML header and attached-block region — the XISFFile entry point skeleton, analogous to opening a FITSFile.

Tasks
  • XISFFile.swift: public convenience init(url:options:) and public init(data:options:).
  • Validate the 8-byte signature XISF0100 (offsets 0–7).
  • Read the 4-byte little-endian header-length field (offsets 8–11); validate the 4-byte reserved field is zero (offsets 12–15).
  • Slice out the UTF-8 XML header (starts at offset 16, length = header-length field) and retain the remaining bytes as the attached-block region (with absolute file offsets preserved for attachment:position:size resolution).
  • Expose the raw XML header string (e.g. public var headerXML: String) and a maxHeaderSize/signature constant, paralleling FITSFile.blockSize.
  • Unit tests: valid preamble, bad signature, truncated file, non-zero reserved, header length past EOF.

2026-06-30

Implementation notes

Implemented with TDD (tests written first, watched fail, then implemented to green). XISFFile lives in SwiftXISF/Core/, mirrored by SwiftXISFTests/Core/XISFFile.swift.

  • XISFFile: public class … : CustomStringConvertible, structured like FITSFile. public convenience init(url:options:) reads the file with .mappedIfSafe and classifies a read failure after the attempt (no TOCTOU gap) into invalidFileURL vs cannotReadFile, exactly as FITSFile does. public init(data:options:) validates the preamble and slices the header.
  • Preamble validation order: empty → dataError; signature mismatch (or a file too short to hold the 8-byte marker) → invalidSignature; file shorter than the 16-byte preamble → invalidHeaderLength; non-zero reserved field (offset 12, read little-endian) → invalidSignature unless lenient; zero header length → invalidHeaderLength; header extending past EOF → invalidHeaderLength; header bytes not valid UTF-8 → malformedXML.
  • Exposed state: public let headerXML: String only — the verbatim UTF-8 header text (typed XML parsing is M4). (Revised after review: an earlier draft also stored public let data: Data; removed — see Review comments.)
  • Constants: public static let signature = "XISF0100" and public static let preambleSize = 16 (parallels FITSFile.blockSize).
  • Decision — no maxHeaderSize constant: the plan mentioned a "maxHeaderSize/signature constant", but the verified XISF 1.0 facts define only a 65-byte minimum header, no maximum. Since the header is sliced from already-resident bytes (no allocation), an over-large length simply fails the past-EOF bounds check — there is no DoS surface to guard. I therefore exposed the signature/preamble constants and did not invent an arbitrary maximum that could reject valid files. Flag for review.
  • Decision — reserved field classification: a non-zero reserved field is reported via invalidSignature (its doc comment was broadened to cover the whole binary preamble: marker + reserved field), and it is tolerated under .lenient (allowSpecDeviations) since it is a harmless deviation. Both behaviors are covered by tests.
  • Decision — header-length minimum: M3 enforces only headerLength > 0; the spec's 65-byte minimum is left to be enforced naturally by the XML parsing in M4 rather than as a magic number here.
  • Xcode project sync: added XISFFile.swift source + test file references and build-phase entries to the Core groups.
  • Verification: swift test → 27 tests pass; xcodebuild … build test → ** TEST SUCCEEDED **, 27 tests pass. Tests cover valid preamble, header parsing independent of trailing attachment bytes, bad signature, empty data, truncated file, non-zero reserved (strict reject / lenient tolerate), header length past EOF, zero header length, and URL read / missing URL.
Review comments
  • Don't keep a top-level whole-file data buffer — SwiftFITS doesn't. Verified by reading the source: FITSBlock stores public let data: Data (each block keeps its own bytes), FITSFile stores sections (which hold the blocks), and FITSFile.data is a computed property that reconstructs the file from those sections. There is no separate whole-file field. The public let data: Data I put on XISFFile is exactly that redundant buffer, and it presupposes lazy post-init attachment resolution.
  • Resolution. Adopt the SwiftFITS model: parse eagerly inside init(data:) so each XISFDataBlock (M7) holds its own resolved byte slice (cheap, mmap/COW-backed), and let any whole-file data be computed if a later milestone needs it. For M3: removed public let data: Data from XISFFile (keeping only headerXML), removed the now-premature retainsAttachmentRegionWithAbsoluteOffsets test, and changed readsFromURL to assert on headerXML. The raw bytes remain reachable as the init-local for when M7/M10 add block/image parsing.

Implementation notes

Implemented with TDD. New internal infrastructure in SwiftXISF/Core/: XISFElement (the element-tree node) and XISFXMLParser (the XMLParser-based walker), with Test_XISFXMLParser and additional Test_XISFFile cases.

  • Decision — namespace handling (confirmed with user: "robust and standard compliant — the XML standard"). XML Namespaces are part of the XML standard, so the walker is namespace-aware: XMLParser.shouldProcessNamespaces = true. Elements are matched by their local name (so a default-namespaced <Image> and a prefixed <x:Image> are the same element), with the namespace URI carried separately on each node. Names are compared case-sensitively, as XML requires.
  • XISFElement (internal): name (local), namespaceURI: String?, attributes: [String:String] (keyed by local name), ordered children, accumulated content (+ trimmedContent), and a children(named:) helper. Not part of the public API.
  • XISFXMLParser (internal): NSObject/XMLParserDelegate with a static parse(_:) throws -> XISFElement that builds the tree via a stack. Well-formedness failures and any delegate parseErrorOccurred surface as XISFError.malformedXML; an empty/rootless document is also rejected.
  • Security: external-entity resolution is left disabled (the XMLParser default), so the parser never reads files outside the header (avoids XXE) — a standards/security best practice, noted in the doc comment.
  • TDD discovery: with namespace processing on, XMLParser reports a no-namespace element's URI as an empty string rather than nil. Normalized that to nil in the walker so "no namespace" has a single representation; a dedicated test pins this.
  • XISFFile wiring: init now parses headerXML into the tree and validates the root via a private validateRoot. Added public static let namespace = "http://www.pixinsight.com/xisf", stored the tree as internal let root, and exposed public var headerElementNames: [String] (the local names of the root's direct children).
  • Root validation rules: local name must be xisf (else invalidElement); the root must be in the XISF namespace or in no namespace (tolerating headers that omit the declaration), while a foreign namespace is rejected; version must be 1.0 under strict, tolerated (missing or different) under .lenient (allowSpecDeviations).

Implementation notes

Implemented with TDD. New types in SwiftXISF/Model/: XISFPropertyType and XISFValue, with Test_XISFPropertyType and Test_XISFValue in SwiftXISFTests/Model/.

  • Spec verification (not memory). The PixInsight spec page 403s to automated fetches, so the exact type strings and value formats were verified against two open-source XISF implementations: libXISF (a complete C++ XISF 1.0 reader/writer — its Variant::Type enum and string tables) and sergio-dr/xisf (a Python decoder). Cross-checked and consistent.
  • Plan correction — no 128-bit complex. The plan's Appendix B shorthand listed Complex32/64/128 and C{32,64,128}Vector. XISF 1.0 has no 128-bit complex (nor 128-bit scalar) type — only Complex32/Complex64 (scalars) and C32/C64 (vector/matrix). Implemented the verified set; a test asserts Complex128/C128Vector/Int128/Float128 are unrecognized.
  • XISFPropertyType: String-raw-valued enum whose raw values are the exact spec type strings — Boolean, Int8/16/32/64, UInt8/16/32/64, Float32/64, Complex32/64, String, TimePoint, ByteArray, the I8…C64 Vector family, and the corresponding Matrix family. CaseIterable, Sendable. A nested Category (scalar/complex/string/timePoint/vector/matrix) classifies how each type's value is carried. Note: Complex32 = two 32-bit float components, Complex64 = two 64-bit (the digits are the per-component width).
  • XISFValue: public enum … : Equatable, Hashable, Sendable, mirroring FITSValue. Cases boolean, integer(Int64) (all signed widths, widened), unsignedInteger(UInt64) (all unsigned widths; needs its own case since UInt64.max exceeds Int64), float(Double), complex(real:imaginary:), string, timePoint(Date); nested Kind with descriptions; per-case accessors. NaN is treated equal to NaN in == (for floats and either complex component) with hash(into:) kept consistent, exactly as FITSValue does.
  • Scalar parsing — XISFValue.value(fromAttribute:type:): trims surrounding whitespace, then by type: Boolean accepts 0/1/true/false (words case-insensitive); integers parse decimal and are range-checked against the declared width via T(String) (so 300 as Int8, or a value overflowing Int64, throws); floats via Double(_:) (accepts scientific notation and nan); complex as (real,imaginary) split on the first comma with each component parsed as Double; time points as ISO 8601 (both with and without fractional seconds — see below). Failures throw XISFError.invalidElement. The value-attribute parser rejects String (carried as element content, handled in M6) and vector/matrix/ByteArray (carried in data blocks).
  • Decision — TimePoint. The spec leaves the time representation implementation-defined; PixInsight/libXISF use ISO 8601 (libXISF parses %Y-%m-%dT%H:%M:%SZ). I parse with ISO8601DateFormatter, trying the fractional-seconds form first then the whole-second form, so both …:05Z and …:05.500Z work. Formatters are created locally (not stored statically) to stay Swift 6 concurrency-clean, since ISO8601DateFormatter is not Sendable.
  • Scope: per the M5 goal, only scalar (value-attribute) values are parsed now; vector/matrix data-block-backed values are completed in M10. XISFPropertyType already enumerates the vector/matrix families so M10 can build on it.

Implementation notes

Implemented with TDD. New types in SwiftXISF/Model/: XISFProperty and XISFFITSKeyword, with their tests in SwiftXISFTests/Model/ and additional Test_XISFFile cases.

  • Spec verification (libXISF): confirmed the element shapes against libXISF — <Property id type value comment format> (scalar/complex/timePoint in the value attribute; String value is the element's character content; vector/matrix via a data block location) and <FITSKeyword name value comment> (empty value for HISTORY/COMMENT).

Unlike FITSProperty (a class that mutates via merge), these are immutable parsed records with no identity, so value types give clean value semantics and free Sendable.   • XISFProperty: id, type (XISFPropertyType), value (XISFValue), optional comment/format.   • Property id validation: XISF property ids are hierarchical, colon-separated (e.g. Observation:Time:Start), not simple identifiers. Under strict parsing the name must be ≤ 8 characters from [A-Z0-9_-]; .lenient tolerates longer/lowercase names. The FITS value is kept verbatim as a string (XISF stores it pre-formatted), not re-typed.

B. Verified XISF 1.0 structural reference

Quoted from the XISF 1.0 specification (pixinsight.com/doc/docs/XISF-1.0-spec) — used to write this plan; reconfirm against the spec while implementing.

Monolithic file layout (byte offsets):

0–7    Signature   ASCII "XISF0100"
8–11   HeaderLen   uint32, LITTLE-ENDIAN  (length of the XML header)
12–15  Reserved    4 bytes, must be zero
16…    XML header  UTF-8, length = HeaderLen   (<?xml …>…</xisf>, min 65 bytes)
…      (optional zero padding, then attached data blocks)

Root: <xisf version="1.0" xmlns="http://www.pixinsight.com/xisf" …>. Core child elements: Image, Property, FITSKeyword, Metadata, Thumbnail, Reference, ICCProfile, RGBWorkingSpace, DisplayFunction, ColorFilterArray, Resolution, Structure/Table (tables out of scope).

Image attributes: geometry="w:h:…:channels"; sampleFormat ∈ {UInt8, UInt16, UInt32, UInt64, Float32, Float64, Complex32, Complex64}; colorSpace ∈ {Gray, RGB, CIELab} (default Gray); pixelStorage ∈ {Planar, Normal} (default Planar); bounds="lo:hi" (required for float formats); imageType, offset, orientation, id ([_a-zA-Z][_a-zA-Z0-9]*), uuid; plus the shared data-block attributes location, compression, checksum, byteOrder.

Data-block location forms:

location="inline:base64"      (data in element character content)
location="inline:hex"         (lowercase hex)
location="embedded"           (child <Data encoding="base64|hex">…</Data>)
location="attachment:POSITION:SIZE"          e.g. attachment:4570:1428362
location="url(URL)"  |  "url(URL):INDEX-ID"  (parens XML-escaped; index-id hex)
location="path(/abs/path)"  |  "path(@header_dir/rel/path):INDEX-ID"

compression forms:

compression="zlib:UNCOMPRESSED-SIZE"           e.g. zlib:6220800
compression="lz4:UNCOMPRESSED-SIZE"
compression="lz4hc:UNCOMPRESSED-SIZE"
compression="zlib+sh:UNCOMPRESSED-SIZE:ITEM-SIZE"   (+sh = byte shuffling)
compression="lz4+sh:UNCOMPRESSED-SIZE:ITEM-SIZE"
compression="lz4hc+sh:UNCOMPRESSED-SIZE:ITEM-SIZE"
subblocks="c1,u1:c2,u2:…"     (optional split-compression sub-blocks)
zstd / zstd+sh                 (same syntax; deferred to M13 — external dep)

checksum form: checksum="algorithm:digest", digest = lowercase base16. Algorithms: sha-1/sha1 (must-support), sha-256/sha256, sha-512/sha512, sha3-256, sha3-512.

Property types: scalars Boolean, Int8…Int64, UInt8…UInt64, Float32/64, Complex32/64/128, String, TimePoint; vectors ByteArray, {I,U}{8,16,32,64}Vector, F{32,64}Vector, C{32,64,128}Vector; matrices the corresponding *Matrix forms. Scalars use value="…"; strings may be inline content; vectors add length="N"; matrices add rows/columns; vector/matrix/large-string values use a data block (location).

FITSKeyword attributes: name (8-char, [A-Z0-9_-], left-justified), value (empty for HISTORY/COMMENT), comment.

C. Compression / digest API mapping (to verify during implementation)

  • zlib, lz4 → Apple Compression (COMPRESSION_ZLIB, COMPRESSION_LZ4); lz4hc streams decode with the LZ4 decoder. Confirm raw vs. framed expectations against real fixtures.
  • sha-256/sha-512 → CryptoKit.SHA256/SHA512; sha-1 → Insecure.SHA1 or CommonCrypto. sha3-* only if a practical implementation is available.
  • zstd → no Apple API; external dependency (M13).
